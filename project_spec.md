# Project Specification — Math Tutor Agent

## Overview

An adaptive K-12 math tutoring agent that presents problems, evaluates student answers with a symbolic solver, retrieves relevant explanatory content via RAG, and adjusts difficulty and topic selection based on a persistent model of student mastery.

**Stack**: Python 3.12 · LangGraph · ChromaDB · SymPy · Ollama (llama3.1:8b) · NiceGUI + KaTeX

---

## Functional Requirements

### FR-1: Problem Presentation
- The system must present one math problem at a time, appropriate to the student's current mastery level and topic queue.
- Problems must be written **in pure mathematical notation** — no word problems, story contexts, or named entities — with one controlled exception: percentages (below). All mathematical content must be valid LaTeX wrapped in `$...$` delimiters and rendered as typeset math in the UI.
  - Arithmetic problems are presented as a bare expression awaiting an answer: `$\frac{1}{6} + \frac{2}{3} =$`
  - Proportions are presented with the unknown: `$\frac{3}{4} = \frac{x}{8}$,  $x = ?$`
  - Equation problems are presented with the unknown: `$3x - 9 = 12$,  $x = ?$`
  - **Percentages** use exactly one of two fixed, well-formed question templates — `"What is $P\%$ of $N$?"` or `"$M$ is what percent of $N$?"`. These deterministic templates are the only allowed lightly-worded form; they map directly to a SymPy computation with no story/answer drift.
- Problems must be generated by an LLM and validated to include a machine-checkable answer. The correct answer must be computed by SymPy from the LLM's expression — never taken from the LLM's own arithmetic.
- Difficulty must be on a 1–5 integer scale where 1 = introductory and 5 = advanced.

### FR-2: Answer Evaluation
- Student answers must be evaluated using symbolic math (SymPy), not an LLM.
- Equivalent forms of the correct answer must be accepted (`1/2`, `0.5`, `2/4` are all correct for the same answer).
- For wrong answers, the system must additionally classify the error into a category (e.g., `sign_error`, `wrong_operation`, `arithmetic_mistake`, `conceptual_error`). This classification is **internal only**: it sharpens knowledge-base retrieval and accumulates in per-topic error statistics, but is never displayed to the student.
- Unparseable student input (non-numeric, ambiguous) must surface a clear parse-error message, not silently mark wrong.

### FR-3: Retrieval-Augmented Feedback with Verified Derivations
- After every answer evaluation — correct or incorrect — the system must show: a result banner, a short LLM concept note, and the full solution derivation.
- **The derivation must be computed symbolically (SymPy), never written by an LLM.** Every displayed equality must pass symbolic verification before it is shown; inputs that cannot be derived granularly fall back to a single verified `expression = result` step. Wrong math in an explanation must be structurally impossible.
- The LLM concept note (2–3 sentences) must be generated from a prompt that injects retrieved knowledge-base content — the LLM must not explain from parametric memory alone, must not perform arithmetic, and must not emit meta-commentary about formatting or sections. Separate prompts are used for correct and incorrect answers.
- Retrieval must filter by `topic`, `subtopic`, and optionally `misconception_tag` to return targeted content.
- All math in the note and derivation must be `$...$` LaTeX, typeset in the UI.

### FR-4: Student State
- The system must persist student state across sessions in `data/students/{student_id}.json`.
- State must include: mastery per topic and per subtopic (float 0.0–1.0), attempt and error-pattern counts, recent problem-shape signatures (a per-subtopic avoid-list), and full session history (attempt records).
- Mastery scores must update after each attempt using an EMA: `new = 0.7 * old + 0.3 * (1.0 if correct else 0.0)`, snapping to 1.0 once a correct answer brings the score within 0.05 of full.

### FR-5: Adaptive Topic and Difficulty Selection
- Selection operates at the subtopic level and favors weakness: a cold-start pick of a never-attempted subtopic, an exploration fraction that keeps coverage broad, and otherwise a weighted-random draw by a weakness priority (low mastery + recent errors).
- Difficulty tracks demonstrated mastery rather than a separate counter: `difficulty = clamp(int(mastery * 5) + 1, 1, 5)`, so it moves with the progress bar.
- A new student with no prior state begins from the cold-start path over the curriculum sequence (fractions_ratios → algebra) at the introductory band.

### FR-6: Session UI
- The UI is a NiceGUI app: a single Python process serving an event-driven web interface, launched with `python src/ui.py`.
- It must show: the current problem (LaTeX typeset via KaTeX), answer input, feedback after submission (LaTeX typeset), current mastery scores per topic (progress bars), and session attempt count.
- It must show a **live operation feed** while the agent works — one line per completed graph node (e.g. "Checking answer (SymPy symbolic solver)", "Searching knowledge base (ChromaDB RAG)") — so the under-the-hood pipeline is transparent.
- The session has two phases: *answering* (problem + input) and *reviewing* (problem + result banner + explanation + "Next problem" button). A new problem is generated only when the student advances.
- The UI must not show the correct answer before or during student input, and must remain responsive during LLM calls.

## Knowledge Base Requirements

### KBR-1: Content Coverage
This is an **arithmetic and algebra** practice platform. Topics in scope:
- `fractions_ratios` (label "Fractions & Ratios"): equivalent fractions, addition/subtraction, multiplication/division, proportions, percentages
- `algebra`: linear equations (one-step through multi-step, both-sides, distribution, fractional), evaluating expressions, linear relationships (slope, y = mx + b)

Geometry is intentionally out of scope — it is hard to pose without visual diagrams. Fractions and ratios are merged into one topic because they are conceptually linked (proportions and percentages are built on fractions).

### KBR-2: Content Types
Each chunk must be one of:
- `concept_explanation` — defines the concept and key properties
- `worked_example` — step-by-step solution to a representative problem
- `common_misconception` — describes a frequent error and why it is wrong

### KBR-3: Chunk Structure
Each chunk stored in ChromaDB must carry the following metadata:

| Field | Type | Description |
|---|---|---|
| `topic` | string | One of: `fractions_ratios`, `algebra` |
| `subtopic` | string | Finer-grained concept within the topic |
| `content_type` | string | `concept_explanation`, `worked_example`, `common_misconception` |
| `difficulty` | int | 1–5 |
| `misconception_tag` | string or null | Matches error categories from FR-2 |

### KBR-4: Scale
Target: ~45 chunks across the two topics. **Depth and clarity are preferred over breadth** — `concept_explanation` and `worked_example` chunks must show every step and explain each one, written for a student unfamiliar with the topic. Every subtopic carries at least one `concept_explanation`, multiple `worked_example`s, and at least one `common_misconception`.

## API Contracts

### Agent Input / Output

The LangGraph graph is invoked with a partial `TutorState`. The UI calls the graph in two phases:

**Phase A — Start turn** (begin new problem):
```python
# Input (partial state)
{
    "student_id": str,          # e.g. "alice_123"
    "student_answer": ""        # empty string signals: generate new problem
}

# Output (full state after the graph pauses before evaluate_answer)
{
    "current_problem": str,     # human-readable problem text
    "current_topic": str,
    "current_difficulty": int,
    "mastery": dict[str, float]
}
```

**Phase B — Submit answer**:
```python
# Input
{
    "student_id": str,
    "student_answer": str,      # student's typed answer
    # ... rest of state carried from Phase A
}

# Output
{
    "evaluation": {
        "is_correct": bool,
        "error_category": str | None,   # None if correct
        "parse_error": bool             # True if SymPy could not parse input
    },
    "feedback": str,            # LLM-generated explanation with retrieved context
    "mastery": dict[str, float] # updated after this attempt
}
```

### `generate_problem` — LLM structured output

The LLM prompt must instruct the model to return JSON conforming to:

```json
{
  "problem_text": "$3x - 5 = 10$,  $x = ?$",
  "sympy_expression": "Eq(3*x - 5, 10)",
  "topic": "algebra",
  "subtopic": "linear_equations",
  "difficulty": 3
}
```

The node parses and repairs this JSON, evaluates `sympy_expression` with SymPy itself to derive the answer, and retries up to 3 times (then a deterministic on-topic fallback) if the response is malformed or does not reduce to a concrete number.

### `symbolic_check` — SymPy evaluation

```python
def symbolic_check(student_input: str, sympy_answer: str) -> bool | None:
    """
    Returns:
      True   — student answer is symbolically equivalent to correct answer
      False  — student answer is wrong
      None   — student input could not be parsed (triggers parse_error in evaluation)
    """
```

Handles:
- Integer and decimal answers: `"5"`, `"2.5"`
- Fractions: `"3/4"`, `"1 1/2"` (mixed numbers need pre-processing to `"3/2"`)
- Simple algebraic expressions: `"x + 1"` (for open-ended expression answers)

### Student State — Pydantic Models

```python
class AttemptRecord(BaseModel):
    timestamp: datetime
    topic: str
    subtopic: str
    difficulty: int
    problem_text: str
    student_answer: str
    is_correct: bool
    error_category: str | None
    parse_error: bool

class TopicMastery(BaseModel):
    topic: str
    mastery_score: float           # 0.0–1.0, EMA-updated
    current_difficulty: int        # 1–5
    attempts: int
    error_pattern_counts: dict[str, int]   # error_category → count
    last_updated: datetime

# SubtopicMastery has the same shape (plus a subtopic field) and is keyed by
# subtopic_key(topic, subtopic). It is the granularity personalization uses.

class StudentState(BaseModel):
    student_id: str
    created_at: datetime
    last_active: datetime
    topic_mastery: dict[str, TopicMastery]
    subtopic_mastery: dict[str, SubtopicMastery]
    recent_signatures: dict[str, list[str]]   # per-subtopic problem-shape avoid-list
    attempt_history: list[AttemptRecord]

    def save(self, base_dir: Path) -> None: ...
    @classmethod
    def load(cls, student_id: str, base_dir: Path) -> "StudentState": ...
    @classmethod
    def load_or_new(cls, student_id: str, base_dir: Path) -> "StudentState": ...
```

## Non-Functional Requirements

| Requirement | Target |
|---|---|
| LLM latency (problem generation) | < 10s on local machine with llama3:8b |
| LLM latency (feedback generation) | < 15s on local machine |
| SymPy evaluation | < 100ms |
| ChromaDB query | < 500ms |
| State persistence (read/write) | < 50ms |
| Knowledge base size | ~45 chunks (depth over breadth) |
| Supported topics | fractions & ratios, algebra |
| Difficulty levels | 1–5 integer scale |

---

## Out of Scope (v1)

- Multi-student sessions (concurrent users)
- Authentication / login
- Proof-based or open-response questions
- Token-level streaming of LLM output to the UI (the op feed streams per-node, not per-token)
- Automated knowledge base generation (all chunks are hand-authored)
- Any cloud deployment — fully local

---

## Build Phases

| Phase | Deliverable | Success Criteria |
|---|---|---|
| 1 — Skeleton | End-to-end working turn | Problem appears → answer submitted → feedback shown → new problem |
| 2 — Knowledge Base | Two topics (fractions & ratios, algebra), ~45 thorough chunks ingested | RAG retrieval returns topically relevant chunks |
| 3 — Adaptive Logic | Mastery tracking + adaptation | 5 correct answers raise mastery; difficulty adjusts |
| 4 — Polish | UI + onboarding + error handling | Full session with diagnostic, progress view, graceful errors |
