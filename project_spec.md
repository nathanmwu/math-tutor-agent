# Project Specification — Math Tutor Agent

## Overview

An adaptive K-12 math tutoring agent that presents problems, evaluates student answers with a symbolic solver, retrieves relevant explanatory content via RAG, and adjusts difficulty and topic selection based on a persistent model of student mastery.

**Stack**: Python 3.11 · LangGraph · ChromaDB · SymPy · Ollama (llama3:8b) · Streamlit

---

## Functional Requirements

### FR-1: Problem Presentation
- The system must present one math problem at a time, appropriate to the student's current mastery level and topic queue.
- Problems must be generated by an LLM and validated to include a machine-checkable answer.
- Difficulty must be on a 1–5 integer scale where 1 = introductory and 5 = advanced.

### FR-2: Answer Evaluation
- Student answers must be evaluated using symbolic math (SymPy), not an LLM.
- Equivalent forms of the correct answer must be accepted (`1/2`, `0.5`, `2/4` are all correct for the same answer).
- For wrong answers, the system must additionally classify the error into a category (e.g., `sign_error`, `wrong_operation`, `arithmetic_mistake`, `conceptual_error`).
- Unparseable student input (non-numeric, ambiguous) must surface a clear parse-error message, not silently mark wrong.

### FR-3: Retrieval-Augmented Feedback
- After every answer evaluation, the system must retrieve relevant content from the knowledge base (concept explanations, worked examples, or misconception notes).
- Retrieval must filter by `topic`, `subtopic`, and optionally `misconception_tag` to return targeted content.
- Retrieved chunks must be injected into the feedback generation prompt — the LLM must not generate explanations from parametric memory alone.

### FR-4: Student State
- The system must persist student state across sessions in `data/students/{student_id}.json`.
- State must include: mastery score per topic (float 0.0–1.0), attempt count per topic, error pattern counts per topic, and full session history (attempt records).
- Mastery scores must update after each attempt using EMA: `new = 0.8 * old + 0.2 * (1.0 if correct else 0.0)`.

### FR-5: Adaptive Topic and Difficulty Selection
- Topic selection must prioritize topics where the student's mastery is lowest (weakest-first), with a 30% chance of introducing a topic the student has not yet attempted (exploration).
- Difficulty must adapt per-topic: increment by 1 on correct answer, decrement by 1 on wrong answer, clamped to [1, 5].
- A new student with no prior state must start at difficulty 2 on the first topic in the curriculum sequence.

### FR-6: Session UI
- The Streamlit UI must show: current problem, answer input, feedback after submission, current mastery scores per topic (progress bars or bar chart), and session attempt count.
- The UI must not show the correct answer before or during student input.

## Knowledge Base Requirements

### KBR-1: Content Coverage
Topics in scope (minimum viable):
- `fractions`: equivalent fractions, addition/subtraction, multiplication/division, mixed numbers
- `ratios`: unit rates, proportions, percentages, scaling
- `algebra`: linear equations, inequalities, substitution, intro to functions
- `geometry`: area and perimeter, Pythagorean theorem, coordinate plane, angle relationships

### KBR-2: Content Types
Each chunk must be one of:
- `concept_explanation` — defines the concept and key properties
- `worked_example` — step-by-step solution to a representative problem
- `common_misconception` — describes a frequent error and why it is wrong

### KBR-3: Chunk Structure
Each chunk stored in ChromaDB must carry the following metadata:

| Field | Type | Description |
|---|---|---|
| `topic` | string | One of: `fractions`, `ratios`, `algebra`, `geometry` |
| `subtopic` | string | Finer-grained concept within the topic |
| `content_type` | string | `concept_explanation`, `worked_example`, `common_misconception` |
| `difficulty` | int | 1–5 |
| `misconception_tag` | string or null | Matches error categories from FR-2 |

### KBR-4: Scale
Target: 200–300 chunks across all topics. Depth (multiple content types per subtopic) is preferred over breadth.

## API Contracts

### Agent Input / Output

The LangGraph graph is invoked with a partial `TutorState`. The UI calls the graph in two phases:

**Phase A — Start turn** (begin new problem):
```python
# Input (partial state)
{
    "student_id": str,          # e.g. "alice_123"
    "student_answer": ""        # empty string signals: generate new problem
}

# Output (full state after graph pause at present_problem node)
{
    "current_problem": str,     # human-readable problem text
    "current_topic": str,
    "current_difficulty": int,
    "mastery": dict[str, float]
}
```

**Phase B — Submit answer**:
```python
# Input
{
    "student_id": str,
    "student_answer": str,      # student's typed answer
    # ... rest of state carried from Phase A
}

# Output
{
    "evaluation": {
        "is_correct": bool,
        "error_category": str | None,   # None if correct
        "parse_error": bool             # True if SymPy could not parse input
    },
    "feedback": str,            # LLM-generated explanation with retrieved context
    "mastery": dict[str, float] # updated after this attempt
}
```

### `generate_problem` — LLM structured output

The LLM prompt must instruct the model to return JSON conforming to:

```json
{
  "problem_text": "Solve for x: 3x - 5 = 10",
  "sympy_answer": "5",
  "topic": "algebra",
  "subtopic": "linear_equations",
  "difficulty": 3
}
```

The node must validate this schema (Pydantic or manual check) and retry up to 2 times if the response is malformed or `sympy_answer` fails `sympify()`.

---

### `symbolic_check` — SymPy evaluation

```python
def symbolic_check(student_input: str, sympy_answer: str) -> bool | None:
    """
    Returns:
      True   — student answer is symbolically equivalent to correct answer
      False  — student answer is wrong
      None   — student input could not be parsed (triggers parse_error in evaluation)
    """
```

Handles:
- Integer and decimal answers: `"5"`, `"2.5"`
- Fractions: `"3/4"`, `"1 1/2"` (mixed numbers need pre-processing to `"3/2"`)
- Simple algebraic expressions: `"x + 1"` (for open-ended expression answers)

### Student State — Pydantic Models

```python
class AttemptRecord(BaseModel):
    timestamp: datetime
    topic: str
    subtopic: str
    difficulty: int
    problem_text: str
    student_answer: str
    is_correct: bool
    error_category: str | None
    parse_error: bool

class TopicMastery(BaseModel):
    topic: str
    mastery_score: float           # 0.0–1.0, EMA-updated
    current_difficulty: int        # 1–5, per-topic
```

## Non-Functional Requirements

| Requirement | Target |
|---|---|
| LLM latency (problem generation) | < 10s on local machine with llama3:8b |
| LLM latency (feedback generation) | < 15s on local machine |
| SymPy evaluation | < 100ms |
| ChromaDB query | < 500ms |
| State persistence (read/write) | < 50ms |
| Knowledge base size | 200–300 chunks |
| Supported topics | fractions, ratios, algebra, geometry |
| Difficulty levels | 1–5 integer scale |

---

## Out of Scope (v1)

- Multi-student sessions (concurrent users)
- Authentication / login
- Proof-based or open-response questions
- LaTeX rendering (plain text problems only in v1)
- Streaming LLM output to UI
- Automated knowledge base generation (all chunks are hand-authored)
- Any cloud deployment — fully local

---

## Build Phases

| Phase | Deliverable | Success Criteria |
|---|---|---|
| 1 — Skeleton | End-to-end working turn | Problem appears → answer submitted → feedback shown → new problem |
| 2 — Knowledge Base | 200+ chunks ingested | RAG retrieval returns topically relevant chunks |
| 3 — Adaptive Logic | Mastery tracking + adaptation | 5 correct answers raise mastery; difficulty adjusts |
| 4 — Polish | UI + onboarding + error handling | Full session with diagnostic, progress view, graceful errors |
